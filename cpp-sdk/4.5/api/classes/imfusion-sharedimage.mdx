`#include <ImFusion/GL/SharedImage.h>`

## Overview

[Image](/cpp-sdk/4.5/api/classes/imfusion-image) shared on multiple devices.

This class aids in resource synchronization between CPU and different GPU APIs. It is constructed with a derived image instance either in memory or one of the graphics APIs, an empty base descriptor class is not allowed. In addition to the built-in representations such as [MemImage](/cpp-sdk/4.5/api/classes/imfusion-memimage) and [GlImage](/cpp-sdk/4.5/api/classes/imfusion-glimage) it also supports custom representations provided by plugins through the [CustomImage](/cpp-sdk/4.5/api/classes/imfusion-customimage) interface.

Internally, for each image representation, a shared pointer is held. Synchronization is fully encapsulated and will be performed even on const instances of this class because mere synchronization between devices does not modify the semantic content of an image.

For individual image access, adhere to the following guidelines:

1. Use descriptor() or [img()](#img) to obtain a const image descriptor regardless of the image location.
2. Use sharedMem(), sharedGl(), and sharedCustom&lt;T&gt;() for getting a shared_ptr to the respective image. If the desired representation is absent or outdated, it will be synchronized, even if the method is called on a const object. Thus, make sure to call these methods only if synchronization is allowed, for instance with a valid [GL](/cpp-sdk/4.5/api/namespaces/imfusion-gl) context when calling sharedGl().
3. The same holds for methods mem(), gl(), and custom&lt;T&gt;(), which return non-owning raw pointers held by sharedMem(), sharedGl(), and sharedCustom&lt;T&gt;() respectively.
4. Use hasMem(), hasGl(), hasCustom&lt;T&gt;() to check for an up-to-date representation while making sure that no synchronization is performed. The returned shared_ptr implicitly casts to bool. Furthermore, it allows for working and to work with the potentially available representation in a thread-safe way. If you use [SharedImage](/cpp-sdk/4.5/api/classes/imfusion-sharedimage) in a multi-threaded environment, do not:
                  if (img.hasGl()) &#123;
                      auto gl = img.gl(); // wrong, could have been overridden in other thread since hasGl()
                      ...
                  &#125;
               but instead:
                  if (auto gl = img.hasGl()) &#123; // correct, guaranteed to return either valid image or nullptr
                      ...
                  &#125;
5. If you change the image in one representation, e.g. in memory, make sure to call [setDirtyMem()](#setdirtymem), [setDirtyGl()](#setdirtygl), [setDirtyCustom&lt;T&gt;()](#setdirtycustom), so that all other representations are marked as outdated.
6. Explicitly call [makeExclusive()](#makeexclusive) to remove all representations except one from this container. This might be useful for memory management, e.g. to free [GL](/cpp-sdk/4.5/api/namespaces/imfusion-gl) textures.

**Warning:** By definition a [GlImage](/cpp-sdk/4.5/api/classes/imfusion-glimage) can not store more than 4 channels (this is a general limitation of OpenGL). [SharedImage](/cpp-sdk/4.5/api/classes/imfusion-sharedimage) will deal with this as follows: When syncing a [MemImage](/cpp-sdk/4.5/api/classes/imfusion-memimage) -&gt; [GlImage](/cpp-sdk/4.5/api/classes/imfusion-glimage), the [GL](/cpp-sdk/4.5/api/namespaces/imfusion-gl) representation will only contain the first 4 channels, any extra channels will be discarded. Subsequent calls to [setDirtyGl()](#setdirtygl) will *not* have any effect but only issue a warning because the [GlImage](/cpp-sdk/4.5/api/classes/imfusion-glimage) is incomplete, and you would lose data otherwise. If you nevertheless want to sync back to [MemImage](/cpp-sdk/4.5/api/classes/imfusion-memimage), you need to do this manually to acknowledge that you may lose data. You can do this for instance by using [GlImage::download()](/cpp-sdk/4.5/api/classes/imfusion-glimage) to create a [MemImage](/cpp-sdk/4.5/api/classes/imfusion-memimage) followed by [SharedImage::assign()](#assign) or [SharedImage::update()](#update-1-2).

**See also:** [Using Image Data](/cpp-sdk/4.5/api/guides/image-data-tutorial)

## Member Enumeration Documentation

### CloneOptions

```cpp
enum ImFusion::SharedImage::CloneOptions
```

Bitset enumeration to configure which aspects of a [SharedImageSet](/cpp-sdk/4.5/api/classes/imfusion-sharedimageset) to copy.

**Enumerator:**

- `Everything` — Perform a deep copy of the image data and shallow copies of the mask and deformation.
- `NoMask` — Do not copy the attached mask.
- `NoDeformation` — Do not copy the attached deformation.
- `ShallowImageCopy` — Do not deep copy the SharedImages but only copy the shared_ptrs.

## Member Data Documentation

### signalChanged

```cpp
Signal <const SharedImage *> ImFusion::SharedImage::signalChanged
```

Called when the content of a [SharedImage](/cpp-sdk/4.5/api/classes/imfusion-sharedimage) has changed (every time one of the setDirty() methods has been called).

**Note:** This method may be called frequently during the processing of algorithms. When connecting to this signal, make absolutely sure to *not* include *any* expensive operations! Otherwise you may introduce a performance hit to the entire application. All you should do is setting a dirty flag yourself if needed.

### signalMaskChanged

```cpp
Signal <const SharedImage *> ImFusion::SharedImage::signalMaskChanged
```

Called when a [Mask](/cpp-sdk/4.5/api/classes/imfusion-mask) object is assigned or removed from the [SharedImage](/cpp-sdk/4.5/api/classes/imfusion-sharedimage).

**Note:** Listeners are not immediately notified when a mask becomes incompatible with the image, but only after the first call to [SharedImage::mask()](#mask).

### m_glImgHasStrippedChannels

*mutable, protected*
